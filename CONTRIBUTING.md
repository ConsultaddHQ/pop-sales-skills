# Contributing to Pop Sales Methodology

The methodology in this repo gets sharper every time someone uses it on a real call and notices what it missed. That's how it improves. You don't need to be technical to contribute — you need to have used the methodology and have something specific to say.

There are four ways to contribute, ranked from easiest to most involved.

---

## Way 1 — Slack #pop-skills (easiest, no GitHub needed)

If you ran `/customer-discovery` or used the Cowork project on a prospect and the methodology got something wrong, missed something, or surprised you in a good way:

Post in **#pop-skills** with:
- The prospect (real or anonymized)
- What the methodology said vs. what actually happened
- What you think should be added or changed

An AE or admin will pick it up, validate it across other recent calls, and update the repo. Your input shapes the methodology even if you never touch GitHub.

This is the path for SDRs and anyone who isn't comfortable with git.

---

## Way 2 — Open an issue (5 minutes, requires GitHub login)

If you have a specific suggestion and want it tracked:

1. Go to https://github.com/ConsultaddHQ/pop-sales-skills/issues
2. Click "New issue"
3. Pick a clear title — "Hidden differentiator framework misses ICP X" beats "general feedback"
4. Describe what's missing and why it matters. Reference a real call if you have one.

Issues get triaged weekly. If your suggestion lands, an admin turns it into a PR or implements it directly.

---

## Way 3 — Edit a file in the GitHub web UI (10 minutes, low-friction code change)

If you spot a typo, want to tighten a sentence, or have a small content edit:

1. Open the file on GitHub
2. Click the pencil icon (top-right) — "Edit this file"
3. Make your change
4. At the bottom, write a short message ("fix: rename Slack channel reference") and click "Propose changes"
5. GitHub creates a PR for you. Done.

An admin reviews and merges. Takes ~24 hours typically.

---

## Way 4 — Pull request from a fork (full workflow, technical contributors)

If you want to add a new reference case, restructure a skill, or make a multi-file change:

1. Fork the repo
2. Clone your fork locally
3. Make your changes on a branch
4. Open a PR back to `main` on `ConsultaddHQ/pop-sales-skills`

PR guidelines:
- One concept per PR. Smaller PRs land faster.
- If adding a reference case, use the existing case files as the template.
- If changing methodology, link to a real call or pattern that motivated the change.
- A short paragraph in the PR description explaining "why this change" beats a 5-page rationale.

---

## What gets accepted

✓ **New reference cases** that demonstrate a prospect archetype not yet covered
✓ **Refinements to existing reference cases** with new insights from later calls
✓ **New objection patterns** observed in the wild that don't fit the existing 5
✓ **New vulnerability signals** with a real example (like the hiring-inflection gate)
✓ **New disqualification triggers** that saved the team time
✓ **Tightening of language** in any document — clarity wins
✓ **New skills** for workflow steps the methodology doesn't cover yet
✓ **Bug fixes in scripts** — the install script especially

✗ **Speculative additions** without evidence from real calls
✗ **Style or formatting changes** with no content benefit
✗ **Adding generic sales advice** that isn't specific to Pop's methodology
✗ **Removing methodology** because "we don't use that anymore" (often the future-you will). Mark it deprecated instead.
